# Scenario 1 — Nmap Port Scan Detection

## 1. Scenario Overview

An attacker performs a network port scan against a target machine in order to identify open services. This is a common early-stage reconnaissance activity. The objective from the Blue Team perspective is to detect the scan and correlate it to an alert.

---

## 2. Objective

- Detect an Nmap port scan initiated from the Kali Linux VM against the Windows or Linux target
- Identify the relevant log entries generated by the scan
- Create or verify a detection rule in Wazuh
- Document the investigation process

---

## 3. Lab Environment

| Component | Detail |
|---|---|
| Attack machine | Kali Linux — 192.168.56.10 (host-only network) |
| Target machine | Windows 10 — 192.168.56.20 |
| Detection platform | Windows Firewall logs, Wazuh (HIDS limitation noted) |
| Network | Isolated host-only adapter — no internet routing |

---

## 4. Simulated Activity

The following commands were run from the Kali Linux VM against the Windows 10 target on the isolated host-only network:

```bash
# Initial scan — host discovery with no ping (Windows Firewall blocks ICMP by default)
nmap -Pn 192.168.56.20

# Full scan with service version detection — Windows Firewall temporarily disabled to reveal open ports
nmap -Pn -sV 192.168.56.20
```

**Results — open ports discovered:**

| Port | State | Service |
|---|---|---|
| 135/tcp | open | Microsoft Windows RPC |
| 139/tcp | open | Microsoft Windows NetBIOS-SSN |
| 445/tcp | open | Microsoft-DS (SMB) |

> All commands executed within the isolated lab on the host-only network (192.168.56.0/24). No external systems were targeted. Windows Firewall was re-enabled immediately after the scan.

---

## 5. Logs Generated

| Log Source | Finding |
|---|---|
| Kali terminal — Nmap output | Host up, 3 open ports identified (135, 139, 445) |
| Wazuh Threat Hunting | No alert generated — see Detection Logic for explanation |

**Key observation:** Wazuh is a HIDS (Host-based Intrusion Detection System). It monitors events on the host machine — process creation, log entries, file changes. Nmap is a network-level attack and does not generate Windows Event Log entries by default, so Wazuh had nothing to alert on. Network scanning detection requires a NIDS (Network IDS) such as Suricata, or Windows Firewall logging to be enabled and forwarded.

---

## 6. Detection Logic

**Wazuh (HIDS limitation):** Wazuh did not generate an alert for the Nmap scan. This is expected — Wazuh monitors host-based events and Windows does not log incoming port scan attempts to the Security Event Log by default.

**To detect Nmap scans in a real environment:**
- Deploy **Suricata** (NIDS) alongside Wazuh — Suricata has built-in Nmap detection signatures
- Enable **Windows Firewall logging** and forward to Wazuh:
  - Enable via: `netsh advfirewall set allprofiles logging droppedconnections enable`
  - Log path: `C:\Windows\System32\LogFiles\Firewall\pfirewall.log`

**Key learning:** A HIDS and NIDS serve different purposes. Both are needed for full coverage.

---

## 7. Investigation Steps

1. Identify the source IP generating the port sweep
2. Review the number of connection attempts in a short time window
3. Determine which ports were targeted and whether any are known sensitive services
4. Check whether the source IP is internal or external to the network
5. Correlate with authentication or process creation events from the same timeframe
6. Determine whether any subsequent suspicious activity followed the scan

## 10. Response / Remediation

- Block the source IP at the network perimeter or firewall if not an authorised scanner
- Alert the relevant team if the scan originated from an internal source not expected to perform scanning
- Review firewall rules to ensure unnecessary ports are not exposed
- Log the incident for tracking

---

## 11. Lessons Learned

- Wazuh is a HIDS — it detects host-based events, not network-level scanning. This is not a gap in Wazuh, it's the correct scope for the tool
- Nmap detection requires network-level monitoring (Suricata, firewall logs) — a critical reason why layered defences matter
- SMB on port 445 being open is a high-value finding — this is the port exploited by EternalBlue/WannaCry
- Windows Firewall blocks ICMP by default which means `-Pn` is needed in Nmap to skip host discovery — attackers know this
- Disabling Windows Firewall revealed the open ports — in a real environment this is why host-based firewalls must be enforced via GPO and not left to individual machines
