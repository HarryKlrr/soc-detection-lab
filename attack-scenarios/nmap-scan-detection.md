# Scenario 1 — Nmap Port Scan Detection

## 1. Scenario Overview

An attacker performs a network port scan against a target machine in order to identify open services. This is a common early-stage reconnaissance activity. The objective from the Blue Team perspective is to detect the scan and correlate it to an alert.

---

## 2. Objective

- Detect an Nmap port scan initiated from the Kali Linux VM against the Windows or Linux target
- Identify the relevant log entries generated by the scan
- Create or verify a detection rule in Wazuh and/or Splunk
- Document the investigation process

---

## 3. Lab Environment

| Component | Detail |

| Target machine | Windows 10 / Ubuntu — 192.168.56.20 or .30 |
| Detection platform | Wazuh, Splunk |
| Network | Isolated host-only adapter — no internet routing |

## 4. Simulated Activity

The following commands were run **only on the local lab network** from the Kali Linux VM:

```
# Example — basic TCP SYN scan against the Windows target (local lab only)
nmap -sS 192.168.56.20

# Example — service version detection
nmap -sV 192.168.56.20
```

> All commands were executed within the isolated lab environment. No external systems were targeted.

## 7. Investigation Steps

1. Identify the source IP generating the port sweep
2. Review the number of connection attempts in a short time window
3. Determine which ports were targeted and whether any are known sensitive services
4. Check whether the source IP is internal or external to the network
5. Correlate with authentication or process creation events from the same timeframe
6. Determine whether any subsequent suspicious activity followed the scan

## 10. Response / Remediation

- Block the source IP at the network perimeter or firewall if not an authorised scanner
- Alert the relevant team if the scan originated from an internal source not expected to perform scanning
- Review firewall rules to ensure unnecessary ports are not exposed
- Log the incident for tracking
